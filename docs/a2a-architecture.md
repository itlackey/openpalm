# A2A Channel Architecture

This document describes how OpenPalm exposes the assistant as an A2A
(Agent-to-Agent) discoverable agent while preserving the Guardian-only
ingress security invariant.

---

## Approach

A2A is implemented as a **channel adapter** (`channel-a2a`) — the same
pattern used by `channel-chat`, `channel-api`, and `channel-discord`.
The channel translates A2A JSON-RPC protocol messages into HMAC-signed
Guardian payloads, and all traffic flows through the Guardian before
reaching the assistant.

### Why a channel adapter (not a second OpenCode instance)?

OpenCode does not yet ship a built-in A2A server. Its draft A2A PR
(#10452) adds client-side support (invoking remote A2A agents via
`@domain.com`), not server-side exposure. Instead, we use the official
[`@a2a-js/sdk`](https://github.com/a2aproject/a2a-js) — the same SDK
OpenCode's own PR depends on — to build a lightweight, spec-compliant
A2A server in a channel adapter.

---

## Architecture

```
External A2A Agent
       │
       │  A2A JSON-RPC (message/send, message/stream, tasks/get)
       ▼
  Caddy :80 → /channels/a2a/*  +  /.well-known/agent.json
       │
       ▼
  channel-a2a :8282            ← @a2a-js/sdk server + Guardian forwarding
       │
       │  HMAC-signed ChannelPayload
       ▼
  Guardian :8080               ← validates signature, nonce, rate limit
       │
       │  OpenCode Server API
       ▼
  Assistant :4096              ← processes message, returns response
```

### Components

| Component | Role |
|---|---|
| `packages/channel-a2a/` | A2A protocol adapter — extends BaseChannel, overrides `start()` to wire SDK |
| `@a2a-js/sdk` | Official A2A JS SDK — JSON-RPC dispatch, task lifecycle, SSE, types |
| `@openpalm/channels-sdk` | HMAC signing (`buildChannelMessage`, `forwardChannelMessage`) |
| `registry/channels/a2a.yml` | Compose overlay for the channel-a2a container |
| `registry/channels/a2a.caddy` | Caddy route snippet for A2A endpoints |

### Key files

```
packages/channel-a2a/
├── package.json
├── src/
│   ├── index.ts          # BaseChannel subclass, overrides start() with SDK wiring
│   ├── executor.ts       # AgentExecutor — bridges A2A events → Guardian forwarding
│   ├── agent-card.ts     # Agent Card definition
│   └── index.test.ts     # Unit tests
```

## A2A Endpoints

| Path | Method | Purpose |
|---|---|---|
| `/.well-known/agent.json` | GET | Agent Card discovery (public, no auth) |
| `/channels/a2a/` | POST | A2A JSON-RPC dispatch (bearer auth) |

### Supported JSON-RPC methods

| Method | Description |
|---|---|
| `message/send` | Send message, wait for complete response |
| `message/stream` | Send message, receive SSE stream of task events |
| `tasks/get` | Poll task status by ID |
| `tasks/cancel` | Cancel a running task |

---

## Authentication

External A2A agents authenticate with a bearer token set via the
`A2A_BEARER_TOKEN` environment variable. The Agent Card discovery
endpoint is public (no auth required) per A2A spec convention.

The channel itself authenticates to the Guardian via HMAC-SHA256 signing
using `CHANNEL_A2A_SECRET` (auto-generated by the admin on install).

---

## Task Lifecycle

The `@a2a-js/sdk` `InMemoryTaskStore` manages the full A2A task lifecycle:

```
submitted → working → completed
                    → failed
                    → canceled (via tasks/cancel)
```

The `OpenPalmExecutor` publishes events through the SDK's `ExecutionEventBus`:

1. **working** — published before forwarding to Guardian
2. **artifact-update** — published with assistant response
3. **completed** — published after Guardian responds successfully
4. **failed** — published on Guardian error or unreachability
5. **canceled** — published on `tasks/cancel` request

---

## Environment Variables

| Variable | Default | Purpose |
|---|---|---|
| `CHANNEL_A2A_SECRET` | (auto-generated) | HMAC secret for Guardian auth |
| `A2A_BEARER_TOKEN` | (empty = no auth) | Bearer token for external A2A agents |
| `A2A_PUBLIC_URL` | (empty) | Public URL for Agent Card `url` field |

## Installation

Install via the admin API (same as any channel):

```bash
curl -X POST http://localhost:8080/admin/channels/install \
  -H "x-admin-token: $ADMIN_TOKEN" \
  -H "content-type: application/json" \
  -d '{"channel": "a2a"}'
```

Or drop `registry/channels/a2a.yml` and `a2a.caddy` into
`$CONFIG_HOME/channels/` and run an update.
